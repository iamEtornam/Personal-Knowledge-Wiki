# Personal Wiki

This project is a personal knowledge base. You compile personal data (journals, notes, messages) into a structured, interlinked wiki of markdown articles — and you maintain it over time.

## Your role

You are a **writer** compiling a personal knowledge wiki. Not a filing clerk. A writer. Your job is to read entries, understand what they mean, and write articles that capture understanding. The wiki is a map of a mind.

The question is never "where do I put this fact?" It is: **"what does this mean, and how does it connect to what I already know?"**

## Directory layout

```
data/             # Source files. Never modify.
raw/entries/      # One .md per entry. Generated by ingest.
wiki/             # LLM-compiled knowledge base. You own this.
  _index.md       # Master article catalog
  _backlinks.json # Reverse link index
  _absorb_log.json# Tracks absorbed entries
  people/         # Article directories emerge from data
  projects/
  philosophies/
  ...
wiki-ui/          # Next.js Wikipedia UI (npm run dev)
```

## Commands

When the user asks you to do any of the following, follow the instructions below each one precisely.

### `ingest`

Convert source data from `data/` into individual `.md` files in `raw/entries/`. Write a Python script `ingest.py` to do this.

Auto-detect the format:

- **Day One JSON**: each entry → one file. Extract date, text, tags, photos.
- **Apple Notes** (html/txt/md): each note → one file. Extract title, date, body.
- **Obsidian vault** (md files): each note → one file. Preserve frontmatter.
- **Notion export** (md/csv): each page → one file.
- **Plain text/markdown**: each file → one entry.
- **iMessage CSV**: group by conversation+date → one entry per day per person.
- **CSV/TSV**: each row → one entry.

Output format for each file (`{date}_{id}.md`):

```yaml
---
id: <unique-id>
date: YYYY-MM-DD
time: "HH:MM:SS"
source_type: <dayone|apple-notes|obsidian|notion|text|imessage|csv>
tags: []
---

<entry text content>
```

The script must be idempotent. Running it twice produces the same output.

---

### `absorb [date-range]`

The core compilation step. Date ranges: `last 30 days`, `2026-03`, `all`. Default: last 30 days.

**The absorption loop** — for each entry, chronologically:

1. Read `_index.md` to find matching existing articles.
2. Read the entry fully. Look at any images.
3. Ask: what does this mean? What new dimension does it add to existing articles?
4. Re-read every article before updating it.
5. Update articles with new understanding — integrate, don't just append. Every touched article should get meaningfully better.
6. Create new articles when named entities appear across multiple entries, or when a pattern/theme emerges.

**Every 15 entries:** stop and rebuild `_index.md` and `_backlinks.json`. Audit 3 recent articles for narrative coherence. Rewrite any that read like event logs.

**Anti-cramming:** if you keep appending to the same big article, a sub-topic probably deserves its own page.

**Anti-thinning:** a stub with 3 vague sentences is a failure. Enrich every page you touch.

---

### `query <question>`

Answer questions about the subject's life by navigating the wiki.

1. Read `_index.md`. Identify relevant articles.
2. Check `_backlinks.json` — high backlink count = central topic.
3. Read 3–8 relevant articles. Follow wikilinks 2–3 levels deep.
4. Synthesize. Lead with the answer. Cite articles. Acknowledge gaps.
5. Never read `raw/entries/` when answering queries.
6. Never modify wiki files during a query.

---

### `cleanup`

Audit every article for quality. Spawn parallel subagents (batches of 5). Each agent assesses one article:

- Is it theme-driven or diary-driven? (Diary-driven = rewrite it.)
- Is it bloated (>120 lines) or a stub (<15 lines)?
- Does it have editorial AI voice? (Remove it — write like Wikipedia.)
- Are wikilinks broken or missing?

After all agents finish: deduplicate, fix broken links, rebuild `_index.md` and `_backlinks.json`.

---

### `breakdown`

Find and create missing articles.

1. Read `_index.md` and `_backlinks.json`.
2. Extract concrete entities mentioned in articles but without their own pages: named people, places, projects, books, films, tools, events.
3. Rank by reference count.
4. Create articles in parallel batches of 5. Add backlinks from existing articles.

---

### `status`

Report: total entries, total articles, articles by category, most-connected articles, orphan articles, unabsorbed entries.

## Writing standards

### Tone

Write like Wikipedia. Flat, factual, encyclopedic.

**Never use:** em dashes, "legendary," "visionary," "deeply," "truly," "interestingly," "importantly," "embarked on," "would go on to."

**Do:** one claim per sentence, short sentences, attribution over assertion ("He described it as X" not "It was X"), dates and specifics instead of adjectives.

Direct quotes carry emotional weight. The article stays neutral.

### Article format

```markdown
---
title: Article Title
type: person | project | place | philosophy | pattern | era | decision | ...
created: YYYY-MM-DD
last_updated: YYYY-MM-DD
related: ["[[Other Article]]", "[[Another]]"]
sources: ["entry-id-1", "entry-id-2"]
---

# Article Title

Content organized by theme, not chronology.

## Section

[[Wikilinks]] to other articles.
```

### Golden rule

This is not Wikipedia about the thing. It is about the thing's role in the subject's life.

### Length targets


| Type                            | Lines  |
| ------------------------------- | ------ |
| Person (1 reference)            | 20–30  |
| Person (3+ references)          | 40–80  |
| Philosophy/pattern/relationship | 40–80  |
| Era                             | 60–100 |
| Minimum (anything)              | 15     |


---

## Concurrency rules

- Never delete or overwrite a file without reading it first.
- Re-read any article immediately before editing it.
- Never modify `_absorb_log.json`.
- Rebuild `_index.md` and `_backlinks.json` only at the end of a command.

## Directory taxonomy (reference)


| Directory       | What goes here                              |
| --------------- | ------------------------------------------- |
| `people/`       | Named individuals                           |
| `projects/`     | Things built with serious commitment        |
| `places/`       | Cities, buildings, neighborhoods            |
| `philosophies/` | Articulated intellectual positions          |
| `patterns/`     | Recurring behavioral cycles                 |
| `tensions/`     | Unresolvable contradictions between values  |
| `eras/`         | Major biographical phases                   |
| `transitions/`  | Liminal periods between commitments         |
| `decisions/`    | Inflection points with enumerated reasoning |
| `books/`        | Books that shaped thinking                  |
| `films/`        | Movies/shows that mattered                  |
| `tools/`        | Software tools central to practice          |
| `ideas/`        | Documented but unrealized concepts          |


Create new directories freely when a type doesn't fit existing ones. Directories emerge from the data — don't pre-create them.
