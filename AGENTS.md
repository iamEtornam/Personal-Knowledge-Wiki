# Personal Wiki — Agent Instructions

This project is a personal knowledge base. You compile personal data (journals, notes, messages, images) into a structured, interlinked wiki of markdown articles and maintain it over time.

## Role

You are a **writer**, not a filing clerk. Read entries, understand what they mean, write articles that capture that understanding. The wiki is a map of a mind. The question is never "where do I put this fact?" but "what does this mean, and how does it connect to what I already know?"

---

## Project layout

```
data/             # Source files (journals, notes, exports). Never modify.
raw/entries/      # One .md per entry. Generated by the ingest step.
wiki/             # Your compiled knowledge base. You own this entirely.
  _index.md       # Master catalog of all articles
  _backlinks.json # Reverse link index: article → articles that link to it
  _absorb_log.json# Tracks which raw entries have been absorbed
  people/         # Article subdirectories emerge from data
  projects/
  philosophies/
  ...             # Create freely as needed
wiki-ui/          # Browser UI. Run: cd wiki-ui && npm run dev
```

## Operations

### INGEST

**Trigger:** user says "ingest", "process my data", or drops files in `data/`.

Write a Python script `ingest.py` that converts files in `data/` into individual `.md` files in `raw/entries/`. Auto-detect format:

| Format | Detection | Handling |
|--------|-----------|----------|
| Day One JSON | `*.json` with `entries` array | One file per entry |
| Apple Notes | `*.html`, `*.txt` exports | One file per note |
| Obsidian | Folder of `*.md` files | One file per note |
| Notion | `*.md` / `*.csv` exports | One file per page |
| iMessage | `*.csv` chat logs | One file per day per conversation |
| Plain text | `*.txt`, `*.md` | One file per file |
| CSV | `*.csv`, `*.tsv` | One row per entry |

Output each entry as `{date}_{id}.md`:
```yaml
---
id: <unique-id>
date: YYYY-MM-DD
time: "HH:MM:SS"
source_type: dayone | apple-notes | obsidian | notion | text | imessage | csv
tags: []
---

<entry text>
```

The script must be idempotent.

---

### ABSORB

**Trigger:** user says "absorb", "compile wiki", "process entries", or specifies a date range.

Process entries in `raw/entries/` chronologically. For each entry:

1. Read `_index.md` to find existing articles that this entry touches.
2. Read the full entry. View any images.
3. Ask: what does this mean? What new dimension does it add?
4. Re-read every article you're about to update.
5. Update articles — integrate, don't append. Every touched article gets richer.
6. Create new articles when entities appear across multiple entries or a theme emerges.

**Checkpoint every 15 entries:**
- Rebuild `_index.md` and `_backlinks.json`
- Check: did you create new articles? If zero in the last 15, you're cramming.
- Re-read 3 recent articles. Rewrite any that read like event logs.

Date ranges: `all`, `last 30 days`, `2026-03`, `2026-03-22`, `2024`.

### QUERY

**Trigger:** user asks a question about themselves, their projects, relationships, history, etc.

1. Read `_index.md`.
2. Check `_backlinks.json` — high backlink count means central topic.
3. Read 3–8 relevant articles. Follow `[[wikilinks]]` 2–3 levels deep.
4. Synthesize an answer. Cite article names. Acknowledge gaps.
5. **Never read `raw/entries/`** when answering — the wiki is the knowledge base.
6. **Never modify wiki files** during a query.

---

### CLEANUP

**Trigger:** user says "cleanup", "audit", "fix the wiki".

Spawn parallel subagents (batches of 5). Each reads one article and:
- Checks for diary-driven structure (bad) vs theme-driven (good)
- Checks for AI editorial voice — strip it, write like Wikipedia
- Checks for broken `[[wikilinks]]`
- Enriches stubs (< 15 lines)
- Splits bloated articles (> 150 lines)

After all agents: fix broken links, rebuild `_index.md` and `_backlinks.json`.

---

### BREAKDOWN

**Trigger:** user says "breakdown", "find missing articles", "expand the wiki".

1. Scan all articles for concrete entities without their own page.
2. Rank by reference count.
3. Create articles in parallel batches of 5.
4. Add backlinks from existing articles to each new one.

---

### STATUS

**Trigger:** user says "status", "stats", "how big is the wiki".

Report: total entries ingested, total absorbed, articles by category, most-connected articles, orphan articles (no inbound links), pending (unabsorbed) entries.

## Writing standards

Write like Wikipedia. Flat, factual, encyclopedic. Never use em dashes, "legendary," "visionary," "deeply," "truly," "interestingly," "importantly," "embarked on," "would go on to," or rhetorical questions. Direct quotes do the emotional work; the article stays neutral.

### Article format

```markdown
---
title: Article Title
type: person | project | place | philosophy | pattern | era | decision | transition | experiment | ...
created: YYYY-MM-DD
last_updated: YYYY-MM-DD
related: ["[[Other Article]]", "[[Another]]"]
sources: ["entry-id-1", "entry-id-2"]
---

# Article Title

Organized by theme, not chronology.

## Section name (theme-driven, not date-driven)

[[Wikilinks]] to related articles.
```

### Golden rule

This is not Wikipedia about the thing. It is about the thing's role in the subject's life.

### Length

Minimum 15 lines. Person with 1 reference: 20–30 lines. Person with 3+ references: 40–80 lines. Philosophy/pattern: 40–80 lines. Era: 60–100 lines.

---

## Concurrency rules

- Always read a file before writing it.
- Re-read every article immediately before editing it.
- Never modify `_absorb_log.json`.
- Rebuild `_index.md` and `_backlinks.json` only at the end of a command, never mid-cycle.
